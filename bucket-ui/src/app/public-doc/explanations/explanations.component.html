Background Information

Terminology

  • Data-Centric Design Lab — The Data-Centric Design Lab is a Delft Design Lab structure, connecting researchers, students and industry partners with the theme of data in Participatory Design. Through its research, the lab members prototype and test open-source digital tools to support the use of data in design.
  • Profile — Profile is a cloud application managing user accounts for the digital tools of the Data-Centric Design Lab. It serves the role of Identity Provider for all mobile and web applications relying on these tools.
  • Person — A Person represents a physical person as a user of the Data-Centric Design Lab's digital tools. They are characterised by a unique username and email address, a name and a password.
  • Bucket — Bucket is a cloud application to facilitate data manipulation and exploration as part of a Design process. It is open-source software written in TypeScript and maintained by the Data-Centric Design Lab.
  • Thing — A Thing represents a physical or virtual component collecting data. For example, a phone which collects acceleration, a website recording number of page views. It is characterised by a unique, auto-generated identifier, a name and a type.
  • Property — A Property represents a variable element of a Thing, generating a data-trace over time. It is composed of dimensions evolving together. For example, a temperature property has one dimension of temperature in Celsius degree while an accelerometer property has three dimensions, one for each axis.
  • Property Type — A Property Type defines the structure of a Property, i.e. the number of Dimensions and the characteristics of these Dimensions.
  • Dimension — A Dimension characterises a data point, including a name, a type (primitive such as string, boolean and number or MIME such as audio/mp3) and a unit (e.g. Km/h or .mp3).
  • Consent — A Consent applies to a Property. It is an explicit agreement from a Person owning a Thing to allow access to the data of one Property of the owned Thing, to a Person, Thing or Group. This consent focuses on access only (no data edition nor property manipulation). This consent can be revoked at all times. It concerns the data collected and updates when new data are collected after consent.
  • Shared Property — A shared Property is a property whose data access has been consented to by the Person who owns it (by owning the Thing it is part of).
  • Group — A group represents a set of Things and Persons. It is owned and managed by a Person. All members of a group can access data of the Property shared with this group.

Infrastructure

The following diagram maps out how things work behind the scene. TU Delft hosts our production instance. We do not have a written tutorial for deploying your own instance, but feel free to reach out if you wish to do so. We'll be happy to help.

At the TU Delft, our software tools are deployed on Linux CentOS virtual machines. Each piece of software runs in a Container orchestrated by Docker Community Edition instances via Docker-Compose.

Open Source, Standing on the Shoulders of Giants

In research we commonly say that we stand on the shoulders of giants as our work is only possible thanks to the many researchers and thinkers before us. Same applies for the open-source community and this project. This platform exists thanks to the minds behinds the software we use.

Relying on open-source software allows us transparency on the running code. It brings robustness and security with many eyes to scrutinies for issues and offer solutions.

Tools

  • Ory - Open Source Identidy infrastructure
  • PostgreSQL - Open Source Relational Database
  • InfluxDB - Open Source Time Series Database
  • Grafana - Open Source Dashboard
  • Envoy - Open Source Edge and Service Proxy server

Frameworks

  • TypeORM - Open Source Object-Relational Mapping (ORM) in Typescript
  • Moscajs/Aedes - Open Source MQTT server

Libraries

  • node-jose - Open Source JS implementation of the JSON Object Signing and Encryption (JOSE)
